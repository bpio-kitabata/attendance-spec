SPECIFICATION v18.0

Jinjer API 連携
自動勤怠チェックシステム 仕様書

設計の基本方針

Jinjer APIから「従業員名簿」と「日次打刻実績」を高速並列取得し、勤怠不備を自動判定します。「日次勤怠」シートに全件を出力しつつ、是正が必要な不備レコードのみを「勤怠不備」シートへ抽出・集約する運用設計です。

Primary Engine
GAS fetchAll (Parallel)
Sync Strategy
Multi-Sheet Sync (洗い替え)
SPECIFICATION 01

従業員名簿取得仕様

ENDPOINT (カテゴリ: 従業員)
GET /v1/employees

社員番号と氏名のマッピングに使用。100名を超える場合も考慮し、ページネーションをループ実行して全件取得します。

項目名キー (JSON)役割
社員番号company.employee_codeマッピングの主軸。
氏名personal.last_name / first_name表示用氏名を生成。
在籍区分enrollment-classification-idパラメータ 0 (在籍) を指定。
SPECIFICATION 02

日次打刻実績取得仕様

ENDPOINT (カテゴリ: 日別データ)
GET /v1/kintai-daily-attendances

指定した日付の全従業員の打刻実績を一括取得。取得対象期間分を並列リクエスト（fetchAll）し、高速に集計します。

項目名キー (JSON)役割
社員番号employee_id名簿Mapとの照合キー。
打刻日dateyyyy-MM-dd。
出勤/退勤時刻attended_at / left_at「hh:mm」形式に整形し判定に使用。
休暇名day_off_request_name休暇区分として出力。
LOGIC

取得対象期間の定義

実行日による動的切り替え

月末月初と月中で、チェック対象範囲を自動的に最適化します。

毎月 1日 ～ 5日

対象: 前月1日 ～ 前月末日

目的: 締め処理に向けた、前月データの最終整合性チェック。

毎月 6日 ～ 末日

対象: 当月1日 ～ 実行日前日

目的: 進行中の当月の打刻不備を即時発見・是正。

SHEET DESIGN

シート構成と役割

1. 日次勤怠シート

※同期のたびに全件を「洗い替え（clearContents）」出力。書式は一切変更しません。

列カラム名役割
A従業員名社員番号に基づき解決された氏名を表示
B日付API取得データ (yyyy-MM-dd)
C-D出退勤時刻HH:mm 形式（打刻なしは「-」）
E休暇区分承認済みの休暇名称
F判定結果シート上の値を元に最新の不備状態を出力

2. 勤怠不備シート

管理者が是正すべき**「現在の不備リスト」**。正常データを除外して抽出します。

※不備がない場合は「現在、勤怠不備はありません。」と表示されます。
